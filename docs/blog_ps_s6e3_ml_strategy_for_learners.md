# 从这个 Kaggle 项目里，最值得学的机器学习策略

如果把这个项目压缩成一条最值得学习的主线，其实并不是“用了多少模型”，而是“如何有纪律地推进一个表格竞赛项目”。

这篇短文只讲重点。

## 0. 学习者推荐实践顺序

如果你想复用这个项目的方法，不要照着 phase 编号硬抄，建议按下面顺序练习：

1. 先做一个能稳定产出 `OOF + submission` 的 baseline。
2. 再补最重要的特征工程，尤其是防泄漏目标编码。
3. 然后再引入不同模型族，观察谁真正带来多样性。
4. 有了稳定 `OOF` 之后，再做 blend 和 stack。
5. 当老模型池接近平台期时，再去找新的低相关底模。

## 1. 先把 baseline 做对，再谈优化

这个项目一开始不是冲着复杂模型去的，而是先做了一个可信的 `XGBoost + leak-free target encoding` 基线。

这样做的意义是：

- 先建立稳定的 `OOF`
- 先保证 submission 可提交
- 先让后面的每次优化都有对照组

很多人做比赛容易一上来就堆模型，但没有稳定 baseline，后面的提升是真是假都说不清。

## 2. 表格题里，特征工程往往比盲目换模型更重要

这个项目的特征工程主线很有代表性：

- 基础缺失值与类型处理
- 防泄漏目标编码
- original-data transfer 特征
- digit / n-gram / 条件统计特征

这里最值得学的是“防泄漏”意识。目标编码如果做错，很容易本地分数很好看，但线上不稳。

所以真正重要的不是“有没有 target encoding”，而是“是不是 leak-free target encoding”。

## 3. 判断模型好坏，不能只看单模分数

这个项目后面越来越重视一件事：模型多样性。

原因很简单：

- 一个模型再强，如果和现有模型太像，融合收益也可能很小
- 一个模型即使不是最强单模，只要误差模式不同，也可能成为有价值的补充

这就是为什么项目后来引入了 `CatBoost`、`LightGBM`、`RealMLP` 等不同模型族。

这里要记住一句话：`OOF` 不只是评估指标，也是后续融合和 stack 的输入基础。

## 4. Stacking 有用，但不是万能药

这个项目里最重要的线上突破来自 `phase10 stack oof v1`，说明二层 stacking 确实能吃到模型之间的互补关系。

但后续也说明了另一件事：

- `phase14` 的本地 OOF 更强
- 可它并没有稳定在线上超过 `phase10`

这提醒我们：

- 本地更高，不一定线上更高
- stack 不是永远继续加复杂度就会继续涨分

当底模池本身没有新增高质量模型时，继续抠二层结构很容易进入边际递减。

## 5. 会止损，和会优化一样重要

这个项目后面做了 `phase15` 和 `phase16` 两条新底模探索路线。

它们并不是完全失败，因为都带来了新认知：

- `phase15` 证明了某套 `orig FE + XGBoost` 方案虽然能跑完，但强度不够
- `phase16` 证明了一些高价值 transfer 信号可能有效，但当前 Kaggle 预算下不够划算

真正成熟的地方在于，项目没有因为“已经投入了很多”就继续硬推，而是及时止损，不把弱底模硬塞进 stack。

## 最后，给学习者的 4 条建议

1. 永远先做一个可信 baseline，再去谈高级技巧。
2. 特征工程要重视防泄漏和分布迁移，不要只会调模型参数。
3. 判断模型值不值得保留，要同时看 `OOF`、线上反馈和多样性。
4. 当一条路线既不够强、又不够省资源时，要敢于停掉它。

如果把这个项目的经验总结成一句话，就是：

做 Kaggle 表格题，真正拉开差距的不是“会不会堆模型”，而是“能不能用稳定验证体系筛出真正值得继续的路线”。
