# PS-S6E3 机器学习策略总结

## 1. 项目目标与当前结论

这个项目围绕 `Kaggle Playground Series - Season 6 Episode 3` 客户流失预测展开，核心目标不是单纯堆模型，而是在可复现工程约束下，持续找到能稳定提升 `ROC AUC` 的策略组合。

截至当前版本，项目已经形成比较清晰的阶段结论：

- 当前最佳 Public LB：`0.91608`
- 最佳线上提交：`phase10 stack oof v1`
- 当前最佳本地 OOF：`0.9184734996`
- 当前本地 OOF 最强路线：`phase14 stronger stack pipeline v1/v2`
- 当前最强单模锚点：`phase8 catboost strong v1`
- 已终止的新底模路线：
  - `phase15_orig_fe_xgb`，远程完整跑通但 `OOF AUC = 0.9129815519`
  - `phase16_catboost_orig_transfer`，压缩到可运行预算后 `OOF AUC = 0.9176360422`

项目目前最重要的认知不是“继续微调现有二层权重”，而是“寻找新的低相关强底模，再接入已经验证有效的 stack 框架”。

## 2. 工程约束如何塑造建模策略

这个项目的机器学习策略和常见本地竞赛仓库不太一样，因为它从一开始就受明确工程约束：

- 正式训练与提交都在 Kaggle 远程执行
- 本地只做最小 smoke test、配置维护、OOF 分析、融合搜索与文档归档
- 任何新路线都必须先证明链路正确，再考虑远程预算和上线价值

这带来几个直接后果：

1. 本地不能把“跑通训练”误认为“策略有效”，本地主要验证脚本链路、输入输出和配置完整性。
2. 远程运行时间预算会反过来限制特征和模型复杂度，尤其影响 `phase15/16` 这类新路线。
3. 项目必须把 `OOF`、`fold AUC`、配置文件、工件落盘做成标准流程，否则无法持续迭代。

因此，这个项目的核心不是单次调参，而是“在预算受限环境中建立一条能持续筛选有效模型的研发流水线”。

## 3. 整体机器学习推进路线

整个项目的实现流程可以压缩成下面这条主线：

`稳定 baseline -> 扩展特征 -> 扩展模型族 -> 做 OOF 受限融合 -> 做二层 stack/hybrid -> 尝试新底模 -> 对弱路线止损`

如果按策略层来拆，可以分成六步：

1. 先建立可信基线，让 `OOF` 和线上提交都可复现。
2. 在基线之上增加稳定的小增益特征，而不是一开始就堆复杂模型。
3. 引入不同模型族，提高误差多样性。
4. 用 `OOF` 去筛选哪些模型值得融合，避免“越融越差”。
5. 用二层 `stacking / hybrid` 吃掉模型间剩余互补关系。
6. 当现有模型池接近平台期时，停止抠权重，转向寻找新的低相关底模。

### 3.1 阶段总表

| 阶段 | 核心策略 | 主要判断依据 | 结果 | 是否继续推进 |
| --- | --- | --- | --- | --- |
| `baseline_v1` | `XGBoost + leak-free target encoding` 建立可信链路 | `OOF`、submission、工件完整性 | 建立稳定起点 | 是 |
| `phase1-5` | 逐步增强特征和模型结构 | 本地验证、`OOF`、是否形成候选底模 | 扩大候选池 | 是 |
| `phase6-9` | 引入 `CatBoost / LightGBM / RealMLP` 等多样性模型并做受限融合 | 强度、多样性、线上反馈 | `phase9` 推动线上结果到 `0.91606` | 是 |
| `phase10` | 首次正式二层 stacking | `OOF` 与 `Public LB` 是否同时支持 | 当前最佳线上锚点 `0.91608` | 是 |
| `phase11-14` | 深化 hybrid 与 stack 特征工程 | `OOF`、线上是否超越 `phase10` | `phase14` 本地 `OOF` 最强，但未线上超越 | 部分保留 |

## 4. 特征工程策略

### 4.1 基础特征处理

基础处理强调稳定和可复现，而不是花哨：

- 数值列统一做类型转换，用训练折中位数补缺
- 类别列统一做缺失填充和类别空间对齐
- 训练、验证、测试三侧的类型处理保持一致

这部分看起来普通，但它是后续所有高级特征的前提。没有稳定的数据类型和缺失处理，`XGBoost`、`CatBoost`、融合和二层学习的结果都不可信。

### 4.2 Leak-free Target Encoding

这是整个项目里最关键的特征工程原则之一。

项目没有直接用全量训练集去计算目标编码，而是坚持：

- 外层 fold 负责模型验证
- 内层 fold 负责在训练折内部生成 OOF 编码

这样做的原因很直接：

- 如果当前样本的编码值间接使用了自己的标签，就会发生泄漏
- 泄漏会让本地分数虚高，但线上收益往往不稳定
- 这个项目把“防泄漏”视为硬约束，而不是可选优化项

从策略层看，`target encoding` 在这里不是为了制造巨大单次提升，而是为了给树模型提供更稳定、更可信的类别统计信号。

### 4.3 Original-data Transfer Features

这个比赛的一个重要现实是：主赛题数据是合成数据，但与经典 Telco 数据分布强相关。因此项目后期逐渐把重点转向“原始数据迁移特征”。

这类特征的核心思路是：

- 不把原始数据当训练集直接拼接
- 而是把原始数据视为外部参考分布
- 从中提炼可迁移的统计信号，再映射回比赛样本

项目中围绕这条线尝试过多种信号：

- 原始数据上的单变量流失率信号
- 条件排序和条件残差信号
- 分位数距离信号
- 分布位置类信号，例如 percentile rank、z-score、gap
- 基于原始类别组合得到的交叉统计特征

这类特征的优点是能把“原始数据的业务结构”迁移到比赛里，缺点是：

- 特征数增长快
- 计算成本上升快
- 很容易和运行时长、内存预算产生冲突

`phase15` 和 `phase16` 都是在这条线上的深入尝试，只是最终在“强度 / 预算 / 可接入性”三者之间没有形成理想平衡。

### 4.4 Digit、N-gram 和交叉统计特征

项目不是只做传统表格特征，也尝试过一些轻量但有区分度的增强：

- 数值文本形态特征，如首位数、尾数、位数
- 类别字段的 bi-gram / tri-gram 组合
- 条件 group 下的 rank、gap、统计量

这些特征体现了一个很重要的方法论：

- 不是所有增益都来自更深模型
- 很多时候，结构简单但分布敏感的特征，也能带来稳定增益

### 4.5 Stack 阶段的 Meta Features

从 `phase10` 到 `phase14`，项目把特征工程扩展到“模型预测本身”。

二层学习使用的并不是单纯的原始预测分数，而是围绕多个底模预测构造出更丰富的 meta features，例如：

- `raw`
- `rank`
- `logit`
- 各模型间统计量
- 与锚点模型的 gap / abs gap
- pairwise absdiff

这一步的本质是：

- 不把多个模型看成简单平均对象
- 而把它们视作一组可以再次做特征工程的输入信号

`phase14` 的价值正体现在这里。它不是单纯换一个二层模型，而是把“预测之间的结构关系”也编码进了二层特征。

## 5. 模型路线演进

### 5.1 Baseline：先把可信链路做出来

项目起点是 `XGBoost + leak-free target encoding`。

这条 baseline 的意义不只是给出一个初始分数，更重要的是建立：

- 标准训练输入
- 标准 OOF 输出
- 标准 submission 输出
- 标准指标记录

只要这条链路是可信的，后续任何新特征、新模型和新融合都能被放回同一坐标系里比较。

### 5.2 早期增强：在树模型骨架上逐步加特征

在 baseline 稳定后，项目逐步进入：

- 更强的特征工程版本
- `Ridge -> XGBoost` 这类结构增强路线
- 更强的单树模型配置

这阶段的关键不是“立刻冲榜”，而是扩大候选模型池，为后续融合做准备。

### 5.3 多样性模型：不是只盯着单模分数

项目真正进入有效冲榜阶段，是在引入不同模型族之后。

关键路线包括：

- `phase6_diverse_tree`：引入 `CatBoost / LightGBM`
- `phase8_catboost_strong`：形成强单模锚点
- `phase9_realmlp_tabm_diverse`：引入低相关神经网络分支

这里最重要的判断标准不是“谁单模最高”，而是：

- 谁足够强
- 谁和已有模型不那么像

`RealMLP` 在这个项目里的价值就很典型。它未必是单模最强，但因为和树模型误差结构不同，所以适合做低权重多样性补丁。

### 5.4 受限融合：把多样性真正转成收益

有了多个底模后，项目没有直接做简单平均，而是先做 OOF 约束下的融合搜索。

这里的核心原则是：

- 先看 `OOF`
- 再看相关性
- 再决定权重和候选池

这一步的目标是避免两个常见误区：

1. 只看单模分数，不看互补性
2. 模型一多就全部硬融

项目在这一步已经拿到过明显线上收益，说明“多样性 + 受限融合”是有效方向。

### 5.5 二层 Stacking / Hybrid：线上收益的关键阶段

`phase10` 到 `phase14` 是项目的第二轮主线。

其中：

- `phase10` 的关键价值，是首次把二层 stacking 明确转化为线上真实增益
- `phase11-13` 继续围绕 hybrid、rank 权重、RealMLP 补丁做探索
- `phase14` 强化了 meta feature 工程，把二层输入做得更完整

这段路线告诉我们两件事：

1. 二层学习在本项目中是有效的，`phase10` 已经证明这一点。
2. 二层学习不是无限增益来源，`phase11-14` 虽然继续抬高本地 OOF，但没有在线上稳定超过 `phase10`

这说明一个常见现象：

- 本地 OOF 更高，不等于线上一定更好
- 当模型池本身没有新增高质量低相关底模时，继续抠二层结构很容易边际递减

### 5.6 新底模探索与止损：phase15 / phase16

当 `phase11-14` 接近平台期后，项目做了正确的策略转向：不再继续死抠二层权重，而是尝试引入新底模。

`phase15_orig_fe_xgb` 的思路是：

- 强化 original-data transfer
- 加入 digit features
- 加入 bi/tri-gram features
- 用 `XGBoost` 做新的低相关强单模

实际结果是：

- 早期版本先遇到远程超时
- 压缩配置后已能完整跑通
- 但最终 `OOF AUC = 0.9129815519`
- 显著低于当前强底模区间

因此它被终止，不进入 stack。

`phase16_catboost_orig_transfer` 的思路是：

- 复用 `phase8` 稳定的 `CatBoost` 骨架
- 用更高保真的 original transfer 信号去重建强单模

实际结果是：

- 多个远程版本先后超时
- 过程中还暴露出“远程未正确读取 config，回退脚本默认配置”的问题
- 修正默认配置与日志后，`v5` 终于完整跑通
- 最终 `OOF AUC = 0.9176360422`

这个分数本身不差，但仍低于 `phase8` 单模锚点，因此结论仍然是：

- 不继续接入 stack
- 保留“某些 original-transfer 信号可能有价值”的认知
- 下一轮应考虑更轻的模型族去承载这类信号

## 6. 训练与验证方法

### 6.1 本地只做最小 smoke

本地 smoke 主要验证三件事：

- 数据采样和脚本参数是否正确
- 训练脚本是否能产出标准工件
- `submission` 是否满足格式要求

它不承担正式模型评估职责。

这意味着项目把“训练正确性”和“策略有效性”拆开了：

- 本地验证训练链路
- Kaggle 远程验证真实效果

### 6.2 交叉验证与 OOF 是研发主坐标

项目训练脚本普遍使用分层 `KFold`，并记录：

- 每折 `AUC`
- `OOF AUC`
- 折间均值和波动
- 测试集预测

这里 `OOF` 的角色非常关键，因为它是：

- 融合搜索的输入
- stack 的训练输入
- 比较不同路线最可复用的统一指标

如果没有高质量 `OOF`，整个融合和二层体系都站不住。

### 6.3 Public LB 负责做外部校验

虽然 `OOF` 是主要研发依据，但项目没有忽视 `Public LB`。

它实际采用的是“双指标判断”：

- 先用 `OOF` 做主要筛选
- 再看 `Public LB` 是否支持这个方向

这也是为什么项目能识别出：

- `phase10` 虽然不是终极最复杂路线，但它在线上最稳定
- `phase14` 本地更强，却没有完成线上超越

### 6.4 工件标准化很重要

每条路线都尽量统一输出：

- `submission`
- `oof_predictions` 或标准化 OOF 文件
- `cv_metrics`
- `run_summary`
- 特征重要性或搜索报告

这件事看起来偏工程，但它直接决定了策略迭代效率。没有统一工件，后续就很难做融合、stack 或路线对比。

## 7. 如何判断一个模型值不值得继续

这个项目形成了比较成熟的判断框架。

先用下面这张快速清单做第一轮筛选：

| 维度 | 要问的问题 | 本项目里的典型结论 |
| --- | --- | --- |
| 强度 | `OOF AUC` 是否进入当前可用强底模区间？ | `phase15` 没进入，因此停止推进 |
| 稳定性 | 折间波动是否可接受？训练能否稳定完成？ | 先解决 `phase16` 配置漂移与超时问题，再谈模型价值 |
| 多样性 | 它和现有强模型是否足够不同？ | `RealMLP` 的价值主要来自低相关补充 |
| 预算 | Kaggle 时长和资源预算下能否稳定跑完？ | 更重的 transfer 版本即使可能更强，也可能不可用 |
| 接入价值 | 值不值得占一个 blend / stack 输入位？ | 弱底模不应硬接入 stack |
| 线上一致性 | `OOF` 变好后，线上是否也支持这个方向？ | `phase14` 本地强，但线上未超过 `phase10` |

### 7.1 第一层：看模型本身是否足够强

至少要看：

- `OOF AUC` 是否进入当前可用模型区间
- 折间波动是否稳定
- 训练时长是否在 Kaggle 预算内

如果模型跑不完，或者即使跑完也明显弱于已有强底模，就没有进入下一阶段的必要。

### 7.2 第二层：看它是否具备互补性

模型值不值得接入融合，不只看单模分数，还要看：

- 和现有锚点是否低相关
- 是否能提供新的错误模式
- 加入候选池后有没有希望提高 stack 或 blend 上限

这是项目从“单模导向”进化到“模型池导向”的关键。

### 7.3 第三层：看它是否值得占用 stack 位置

stack 的位置很宝贵。

一个弱底模接入 stack，可能带来的不是增益，而是噪声。`phase15` 和 `phase16` 最终没有被推进到 stack，核心原因就是：

- 它们没有证明自己比现有锚点更强
- 也没有证明自己已经强到值得占一个 stack 输入位

### 7.4 第四层：看这条路线是否已到止损点

一个项目成熟的重要标志，是知道什么时候停止。

这个仓库在 `phase15/16` 上体现出的判断很健康：

- 不是因为“已经投入很多”就继续硬推
- 而是根据 `OOF`、线上预算和接入价值做止损

这能显著减少无效迭代。

## 8. 项目的优化策略

### 8.1 先做稳定锚点，再做增量

项目一直遵循“先有锚点，再谈优化”：

- 先有 baseline
- 再有强单模
- 再有 blend
- 再有 stack

这样每次优化都有参照物，不会迷失在随机波动里。

### 8.2 优先做低相关增量，而不是抠同类模型细节

当一个模型族已经被压榨得比较充分时，再继续在同类模型上抠小参数，收益往往不如引入低相关新模型。

这也是为什么项目从 `phase11-14` 的二层微调，转向 `phase15/16` 的新底模探索。

即使 `phase15/16` 结果不理想，这个“转向动作”本身仍然是正确的。

### 8.3 把运行预算当成真实约束

项目没有把“理论上更强”当成可落地方案。

`phase16` 给出的一个很重要经验是：

- 某些高保真 original-transfer 版本也许更强
- 但如果在 Kaggle 预算下无法稳定运行，就不是当前阶段的可用解

这说明优化策略必须同时考虑：

- 模型强度
- 资源成本
- 结果可复现性

### 8.4 发现配置漂移和性能瓶颈后及时修正

项目在 `phase16` 中途定位到两个很实际的问题：

- Kaggle 远程没有按预期读取同目录配置
- DataFrame 碎片化会拖慢预处理并制造噪声

这些修正不直接提高 leaderboard，但它们提升了“研发系统本身”的质量。对长期项目来说，这类优化非常重要。

## 9. 当前最优认知与下一步建议

结合整个项目演进，当前最可靠的结论是：

1. `phase10 stack oof v1` 仍然是当前线上最可信锚点。
2. `phase14 stronger stack pipeline` 仍然是当前本地 OOF 最强框架。
3. `phase8 catboost strong` 仍然是非常关键的单模锚点。
4. `phase15` 与 `phase16` 提供了有价值的失败经验，但暂不值得接入 stack。

下一轮更合理的方向不是继续抠 `phase11-14` 的二层权重，而是：

- 新增真正低相关、且能在 Kaggle 预算下跑完的强底模
- 复用 `phase10-14` 已验证有效的融合与 stack 管线
- 继续保留 original-data transfer 的高价值信号，但换更适合承载它们的模型族

如果只用一句话概括这个项目的方法论，那就是：

先做可信基线，再做有效多样性；先用 `OOF` 筛选，再用线上验证；当一条新路线不能成为“强底模”时，及时止损，不把弱模型硬塞进 stack。
